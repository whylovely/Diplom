# Описание архитектуры клиентского приложения (подробно)

Это приложение — программа для учёта личных финансов. Пользователь может вести счета, записывать доходы и расходы, смотреть отчёты и управлять долгами. Ниже — максимально подробное объяснение того, как устроен код «изнутри».

---

## Часть 1. Что вообще за технологии используются?

### 1.1. Язык программирования — C#

Весь код написан на языке **C#** (версия 12, платформа **.NET 8**). Это современный, строго типизированный, объектно-ориентированный язык от Microsoft. Он компилируется в промежуточный код (IL), который затем выполняется средой .NET на любой поддерживаемой платформе (Windows, macOS, Linux).

### 1.2. UI-фреймворк — Avalonia UI (версия 11.3)

Для создания графического интерфейса (окна, кнопки, таблицы, поля ввода) используется фреймворк **Avalonia UI**. 

**Почему именно Avalonia, а не WPF или WinForms?**
- WPF и WinForms работают **только** на Windows.
- Avalonia — кроссплатформенный: одно и то же приложение можно скомпилировать и запустить на Windows, macOS и Linux без изменения кода.

**Как описывается интерфейс?**
Интерфейс описывается в файлах с расширением `.axaml` (по сути это XML-разметка). Это похоже на HTML в веб-разработке: вы описываете структуру элементов (кнопки, текстовые поля, таблицы), а их внешний вид настраивается через стили (аналог CSS).

**Пример из нашего проекта** (файл `MainWindow.axaml`):
```xml
<Border Grid.Row="0" Background="#080C1A" Padding="24,20,24,10">
  <ContentControl Content="{Binding Current}" />
</Border>
```
Здесь `Border` — это контейнер с тёмным фоном (`#080C1A`), а внутри него `ContentControl`, который отображает текущую страницу приложения. Запись `{Binding Current}` означает: «покажи то, что сейчас лежит в свойстве `Current` в коде».

### 1.3. Архитектурный паттерн — MVVM

Приложение построено по паттерну **MVVM** (Model — View — ViewModel). Это способ организации кода, при котором:

- **Model (Модель)** — это данные. Например, класс `Account` содержит информацию о банковском счёте: название, валюта, баланс. Модель ничего не знает о кнопках и окнах.
- **View (Представление)** — это то, что видит пользователь: окна, кнопки, таблицы. View ничего не знает о бизнес-логике. Оно просто отображает данные и передаёт действия пользователя (нажал кнопку) дальше.
- **ViewModel (Вью-Модель)** — это «посредник» между Model и View. Она берёт данные из модели, подготавливает их для отображения и обрабатывает команды пользователя (нажал «Удалить» → ViewModel вызывает удаление).

**Зачем такое разделение?**  
Представьте, что вся логика (подсчёт баланса, сохранение данных, отображение кнопок) была бы в одном файле. При 1000+ строк кода это превращается в кашу. MVVM разбивает код на маленькие, понятные части, каждая из которых отвечает за что-то одно.

**Аналогия из жизни:**
- **Model** — это продукты в холодильнике (сырые данные).
- **ViewModel** — это повар, который берёт продукты, готовит из них блюдо и подаёт на тарелке.
- **View** — это тарелка и стол, на которых красиво лежит еда. Тарелка не знает рецепт, повар не знает, какая тарелка будет использована.

### 1.4. CommunityToolkit.Mvvm (версия 8.2)

Это библиотека от Microsoft, которая **автоматически генерирует** шаблонный код для MVVM. Без неё пришлось бы писать десятки строк повторяющегося кода для каждого свойства. 

**Пример без библиотеки** (пришлось бы писать вручную):
```csharp
private string _name = "";
public string Name
{
    get => _name;
    set
    {
        if (_name != value)
        {
            _name = value;
            OnPropertyChanged(nameof(Name)); // Уведомляем UI, что значение изменилось
        }
    }
}
```

**Пример с CommunityToolkit** (всего одна строка!):
```csharp
[ObservableProperty] private string _name = "";
```

Компилятор **сам** создаст публичное свойство `Name` со всей логикой уведомлений. Это называется **кодогенерация** (Source Generators) — код создаётся автоматически во время компиляции.

## Часть 3. Как приложение запускается (пошагово)

### Шаг 1: `Program.cs` — Точка входа

Когда вы запускаете приложение (командой `dotnet run` или кнопкой в IDE), операционная система вызывает метод `Main()` в файле `Program.cs`:

```csharp
[STAThread]
public static void Main(string[] args) => BuildAvaloniaApp()
    .StartWithClassicDesktopLifetime(args);
```

**Что здесь происходит?**
1. `BuildAvaloniaApp()` — создаёт и настраивает движок Avalonia (подключает тему, шрифты, определяет платформу — Windows/Mac/Linux).
2. `StartWithClassicDesktopLifetime(args)` — запускает приложение в режиме обычного десктопного окна.

### Шаг 2: `App.axaml.cs` — Инициализация

После запуска Avalonia вызывает метод `OnFrameworkInitializationCompleted()` в файле `App.axaml.cs`:

```csharp
public override void OnFrameworkInitializationCompleted()
{
    var win = new MainWindow
    {
        DataContext = new MainWindowViewModel()
    };
    desktop.MainWindow = win;
    MainWindow = win;
}
```

**Что здесь происходит?**
1. Создаётся главное окно приложения (`MainWindow`).
2. К этому окну привязывается его «мозг» — `MainWindowViewModel`. Это и есть связь между View и ViewModel.
3. Окно показывается пользователю.

### Шаг 3: `MainWindowViewModel` — Создание всех страниц

В конструкторе `MainWindowViewModel` происходит самое важное — создаются **все** сервисы и **все** страницы приложения:

```csharp
public MainWindowViewModel()
{
    // 1. Создаём сервисы (источники данных, уведомления, настройки, API)
    _localDb = new LocalDbService();         // Локальная SQLite БД
    _api = new ApiService("http://localhost:5000"); // HTTP клиент к серверу
    _sync = new SyncService(_api, _localDb); // Сервис синхронизации
    _notify = new NotificationService();     // Сервис уведомлений
    _settings = new SettingsService();       // Настройки пользователя

    // 2. Создаём страницы и передаём им сервисы
    AccountsVm = new AccountsViewModel(_localDb, _sync, _notify, ...);
    JournalVm = new JournalViewModel(_localDb);
    NewTxVm = new NewTransactionViewModel(_localDb, _notify, ...);
    ReportVm = new ReportViewModel(_localDb);
    CategoriesVm = new CategoriesViewModel(_localDb, _notify, ...);
    ObligationsVm = new ObligationsViewModel(_localDb, _notify);
    SettingsVm = new SettingsViewModel(_settings, _sync);

    // 3. По умолчанию показываем страницу «Счета»
    _current = AccountsVm;
}
```

**Аналогия:** Представьте, что `MainWindowViewModel` — это директор ресторана. При открытии ресторана он:
1. Нанимает поваров (создаёт сервисы).
2. Открывает залы (создаёт страницы).
3. Направляет первых гостей в главный зал (устанавливает `_current = AccountsVm`).

## Часть 4. Как работает навигация между страницами

### Что видит пользователь?
Внизу экрана есть панель навигации (стилизованная под macOS Dock) — полоса с иконками. Нажимая на иконку, пользователь переключается между разделами (Счета, Категории, Журнал и т.д.).

### Как это работает «под капотом»?

**В `MainWindow.axaml`** есть ключевой элемент:
```xml
<ContentControl Content="{Binding Current}" />
```

Этот элемент — как «рамка для картины». В неё можно вставить любую «картину» (страницу). Запись `{Binding Current}` означает: «показывай то, что лежит в свойстве `Current`».

**В `MainWindowViewModel`** есть свойство `Current`:
```csharp
[ObservableProperty]
private ViewModelBase _current;
```

Когда значение `Current` меняется, UI **автоматически** обновляется и показывает новую страницу.

**Как происходит переключение при нажатии кнопки?**

В `MainWindowViewModel` есть методы для каждой страницы:
```csharp
[RelayCommand] private void NavigateAccounts()   { Current = AccountsVm;   ActivePage = "Accounts"; }
[RelayCommand] private void NavigateJournal()    { Current = JournalVm;    ActivePage = "Journal"; }
[RelayCommand] private void NavigateReport()     { Current = ReportVm;     ActivePage = "Report"; }
// ...и так далее для каждой страницы
```

Атрибут `[RelayCommand]` автоматически создаёт команду `NavigateAccountsCommand`, которую можно привязать к кнопке в XAML:
```xml
<Button Command="{Binding NavigateAccountsCommand}" />
```

**Как Avalonia знает, какой View показать для какого ViewModel?**

В `MainWindow.axaml` есть блок `DataTemplates` — это таблица соответствий:
```xml
<DataTemplate DataType="vm:AccountsViewModel">
  <views:AccountsView/>     ← Если Current = AccountsViewModel, покажи AccountsView
</DataTemplate>
<DataTemplate DataType="vm:JournalViewModel">
  <views:JournalView/>      ← Если Current = JournalViewModel, покажи JournalView
</DataTemplate>
```

**Весь процесс по шагам:**
1. Пользователь нажимает иконку «Журнал» в нижней панели.
2. Кнопка вызывает команду `NavigateJournalCommand`.
3. Команда выполняет код: `Current = JournalVm`.
4. Свойство `Current` изменилось → UI получает уведомление.
5. `ContentControl` смотрит: «Current теперь содержит `JournalViewModel`».
6. Находит в `DataTemplates` соответствие: `JournalViewModel` → `JournalView`.
7. Показывает `JournalView` на экране.

## Часть 5. Модели данных — «Что хранится»

Модели — это простые классы, которые описывают данные. Это как таблицы в базе данных, только в виде C#-классов.

### `Account.cs` — Банковский счёт
```csharp
public sealed partial class Account : ObservableObject
{
    public Guid Id { get; init; }           // Уникальный идентификатор (e.g. "a3b8d1b6-...")
    private string _name = "";              // Название ("Основной счёт", "Наличные")
    public string CurrencyCode { get; set; } = "RUB";  // Валюта (RUB, USD, EUR...)
    public decimal InitialBalance { get; set; }         // Начальный баланс
    private decimal _balance;               // Текущий баланс
    public AccountType Type { get; set; }   // Тип: Актив / Доход / Расход / Долг
    public bool IsMultiCurrency { get; set; } // Многовалютный ли?
    public string? SecondaryCurrencyCode { get; set; } // Вторая валюта (если многовалютный)
    private decimal _secondaryBalance;       // Баланс во второй валюте
}
```

### `Transaction.cs` — Транзакция (операция)
```csharp
public sealed class Transaction
{
    public Guid Id { get; init; }            // Уникальный ID
    public DateTimeOffset Date { get; set; } // Дата операции
    public string Description { get; set; }  // Описание ("Зарплата", "Покупка продуктов")
    public List<Entry> Entries { get; set; } // Список проводок (кто кому сколько)
}
```

Каждая транзакция состоит из **проводок** (`Entry`). Например, транзакция «Покупка продуктов на 500₽» будет иметь две проводки:
- Списание 500₽ со счёта «Наличные» (Кредит)
- Зачисление 500₽ на счёт «Продукты (расходы)» (Дебет)

Это принцип **двойной записи** из бухгалтерии.

### Другие модели:
| Модель           | Что хранит                                                   |
|------------------|--------------------------------------------------------------|
| `Category`       | Категория расходов/доходов (Еда, Транспорт, Зарплата)       |
| `Obligation`     | Долг/обязательство (кто должен, сколько, когда платить)      |
| `Entry`          | Одна проводка (привязана к счёту и транзакции)               |
| `JournalRow`     | Строка в журнале операций (для отображения в таблице)        |
| `UserSettings`   | Настройки пользователя (базовая валюта, первый ли запуск)    |
| `Money`          | Вспомогательный класс для работы с денежными суммами         |

## Часть 6. Сервисы — «Как данные получаются и сохраняются»

Сервисы — это классы, которые выполняют «грязную работу»: читают и записывают данные, показывают всплывающие окна, сохраняют настройки.

### 6.1. `IDataService` — Интерфейс для работы с данными

Это **контракт** (интерфейс), который говорит: «любой сервис данных должен уметь делать вот это»:

```csharp
public interface IDataService
{
    IReadOnlyList<Account> Accounts { get; }        // Получить список счетов
    IReadOnlyList<Category> Categories { get; }     // Получить список категорий
    IReadOnlyList<Transaction> Transactions { get; } // Получить список транзакций
    IReadOnlyList<Obligation> Obligations { get; }   // Получить список долгов

    void AddAccount(Account account);             // Добавить счёт
    void RemoveAccount(Guid id);                  // Удалить счёт
    Task PostTransactionAsync(Transaction tx);    // Провести транзакцию
    // ...и другие методы
}
```

**Зачем нужен интерфейс, а не просто класс?**

Представьте, что вы хотите поменять способ хранения данных. Сейчас данные хранятся в памяти (фейковые). Завтра вы хотите хранить их в базе SQLite. Послезавтра — получать с сервера через интернет.

Если бы весь код был привязан к конкретному классу `MockDS`, при замене пришлось бы переписывать десятки файлов. А с интерфейсом — достаточно **создать новый класс**, который реализует тот же `IDataService`, и **подменить одну строку** в `MainWindowViewModel`:

```csharp
// Было (фейковые данные):
_data = new MockDS();

// Стало (реальная база данных SQLite):
_data = new LocalDbService();
```

Приложение продолжает работать без изменений, потому что оно знает только про интерфейс `IDataService`.

### 6.2. `LocalDbService` — Настоящая база данных (SQLite)

На данный момент фейковый `MockDS` был успешно заменен на `LocalDbService`.
В отличие от памяти, данные сохраняются в файл `finance.db` на компьютере пользователя с помощью **Entity Framework Core (SQLite)**.
Так данные не теряются при закрытии приложения.

### 6.3. `SyncService` и `ApiService` — Связь с сервером

- **`ApiService`** — Общается с API сервера (отправляет HTTP-запросы `GET`, `POST`, `PUT`, `DELETE`).
- **`SyncService`** — Содержит логику синхронизации: скачивает данные с сервера, разрешает конфликты и сохраняет их в `LocalDbService`, обеспечивая работу клиента в офлайн-режиме с актуальными данными после синхронизации. Поддерживает "двустороннюю" синхронизацию (как Pull с сервера, так и Push на сервер).

### 6.3. `NotificationService` — Сервис уведомлений

Отвечает за показ всплывающих окон пользователю:

```csharp
public interface INotificationService
{
    Task ShowErrorAsync(string message, string title = "Ошибка");
    Task ShowWarningAsync(string message, string title = "Предупреждение");
    Task ShowInfoAsync(string message, string title = "Информация");
    Task<bool> ShowConfirmAsync(string message, string title = "Подтверждение");
}
```

Когда, например, пользователь пытается удалить счёт, ViewModel вызывает:
```csharp
bool confirmed = await _notify.ShowConfirmAsync("Удалить счёт 'Наличные'?");
if (confirmed) { /* удаляем */ }
```

Сервис создаёт диалоговое окно (`ConfirmDialog`) и ждёт ответа пользователя — нажмёт он ОК или Отмена.

### 6.4. `SettingsService` — Сервис настроек

Сохраняет и загружает пользовательские настройки (базовая валюта, первый ли это запуск) в JSON-файл на диске:

```
C:\Users\<имя>\AppData\Roaming\Diplom\user_settings.json
```

При первом запуске приложение показывает диалог выбора валюты. После выбора `SettingsService` сохраняет её в файл и больше не показывает этот диалог.

## Часть 8. Views — «Как выглядит каждая страница»

Views — это XAML-файлы, которые описывают расположение элементов на экране. Каждый View привязан к своему ViewModel через механизм Binding.

### Основные View файлы:

| View файл                | Что отображает                                                |
|--------------------------|---------------------------------------------------------------|
| `MainWindow.axaml`       | Главное окно: область контента + нижняя навигационная панель  |
| `AccountsView.axaml`     | Таблица счетов с кнопками добавления и удаления               |
| `CategoriesView.axaml`   | Таблица категорий с кнопками                                  |
| `JournalView.axaml`      | Таблица истории операций с полем поиска                        |
| `NewTransactionView.axaml`| Форма: тип операции, сумма, счёт, категория, описание        |
| `ReportView.axaml`       | Несколько вкладок с таблицами и графиками                     |
| `ObligationsView.axaml`  | Таблица долгов с кнопками управления                          |
| `SettingsView.axaml`     | Выбор валюты и прочие настройки                               |

### DialogViews — Всплывающие окна:

## Часть 11. Ключевые концепции кода

### 11.1. `[ObservableProperty]` — Автоматическое свойство с уведомлением UI

Когда вы пишете:
```csharp
[ObservableProperty] private string _name = "";
```
Компилятор **автоматически** генерирует свойство `Name`, которое при изменении значения отправляет сигнал интерфейсу: «Эй, значение изменилось, перерисуй текст!».

### 11.2. `[RelayCommand]` — Автоматическая команда из метода

Когда вы пишете:
```csharp
[RelayCommand] private void NavigateAccounts() { Current = AccountsVm; }
```
Компилятор создаёт свойство `NavigateAccountsCommand` типа `ICommand`, которое можно привязать к кнопке в XAML.

### 11.3. `{Binding ...}` — Привязка данных

В XAML запись `{Binding Name}` означает: «возьми значение свойства `Name` из привязанного ViewModel и покажи его здесь». Если значение изменится — текст на экране обновится автоматически.

### 11.4. `async/await` — Асинхронность

Многие методы помечены как `async` и содержат `await`. Это нужно, чтобы приложение **не зависало** при длительных операциях (чтение файла, сетевой запрос). Пока операция выполняется, интерфейс остаётся отзывчивым — кнопки нажимаются, окна перемещаются.
