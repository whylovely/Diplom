# Архитектура клиентского приложения

Клиент — десктопное приложение для учёта личных финансов. Пользователь ведёт счета, записывает доходы и расходы, смотрит отчёты и управляет долгами. Данные хранятся локально в SQLite, опционально синхронизируются с сервером.

## 3. Слои приложения

Приложение построено на **repository pattern**. Зависимости направлены строго сверху вниз:

```
ViewModels  →  Services (IDataService)  →  Repositories  →  Data (SQLite)
                   ↑
                Models
```

- **ViewModel** никогда не обращается к репозиторию или SQL напрямую — только через `IDataService`
- **Services** содержат бизнес-логику и оркестрацию
- **Repositories** занимаются только SQL-доступом к одной сущности
- **Data** — технический уровень: соединение, схема, демо-данные

## 4. `Client/Data/` — низкий уровень SQLite

Три класса без зависимостей от доменных моделей:

| Файл | Назначение |
|------|------------|
| `SqliteConFactory.cs` | Единственный источник `ConnectionString` и пути к БД. `Open()` возвращает открытый `SqliteConnection`. Статический `Exec(conn, sql, params[])` — параметризованные команды |
| `DbInitializer.cs` | `EnsureCreated()` — создаёт все таблицы при первом запуске. `MigrateSchema()` — добавляет новые колонки к существующей БД без потери данных |
| `DbSeeder.cs` | `SeedIfEmpty()` — наполняет пустую БД демо-счетами, категориями и транзакциями. Не трогает уже существующие данные |

---

## 5. `Client/Repositories/` — доступ к данным

Каждый репозиторий отвечает ровно за одну сущность. Все следуют единому шаблону:
- Принимают `SqliteConFactory` в конструкторе
- Хранят данные в `List<T> _items`, отдают наружу `IReadOnlyList<T> All`
- Объявляют `public event Action? Changed` и вызывают его после любого изменения
- Метод `Load()` перечитывает всё из БД

| Файл | Класс | Ответственность |
|------|-------|-----------------|
| `AccRepo.cs` | `AccountRepository` | CRUD счетов и групп. `CreateTechnicalAccountsForCategory` — создаёт пару технических счетов («Расходы: X» / «Доходы: X») при добавлении категории. `RebuildTechnicalAccountMappings` — восстанавливает словари `expenseAccByCategory` / `incomeAccByCategory` после загрузки |
| `CategoryRepo.cs` | `CategoriesRepository` | CRUD категорий расходов и доходов |
| `TxRepo.cs` | `TransactionsRepository` | Загрузка транзакций с проводками (Entry). `AddToCache(tx)` — добавляет транзакцию в оперативный список без лишнего SQL. `BuildStorno(id, accounts)` — строит сторно-транзакцию (не сохраняет) |
| `ObligationRepo.cs` | `ObligationRepository` | CRUD долгов и займов, пометка как оплаченных |
| `TemplateRepo.cs` | `TemplatesRepository` | CRUD шаблонов транзакций |
| `CurrencyRepo.cs` | `CurrencyRatesRepository` | Хранит курсы валют к базовой. `GetRate(from, to)` |

## 7. `Client/Services/` — бизнес-логика

### 7.1. Фасад над данными

#### `IDataService` / `LocalDbService`
Единственный публичный контракт для ViewModel. `LocalDbService` создаёт все репозитории, вызывает `Load()`, подписывается на события `Changed` и ретранслирует их через `event Action? DataChanged`.

Оркестрирует операции, затрагивающие несколько репозиториев:

| Метод | Что делает |
|-------|-----------|
| `AddCategory(cat)` | `_categoriesRepo.Add` + `_accountsRepo.CreateTechnicalAccountsForCategory` |
| `PostTransactionAsync(tx)` | SQL-транзакция по Transactions/Entries/Accounts → `_transactionsRepo.AddToCache(tx)` |
| `StornoTransactionAsync(id)` | `_transactionsRepo.BuildStorno(id, accounts)` → `PostTransactionAsync` |
| `ReplaceAllData(...)` | Полная замена данных при Pull-синхронизации |
| `ClearDatabase()` | Удаляет файл БД → пересоздаёт схему → `LoadAll()` |

### 7.2. Сервисы транзакций

| Файл | Назначение |
|------|------------|
| `TransactionBuilder.cs` | Строит `List<Entry>` для всех `TxKindChoice`. Бросает `InvalidOperationException` при бизнес-ошибках (разные валюты при переводе, нет технического счёта) |
| `TransactionValidator.cs` | Проверяет заполненность формы. Возвращает `string? error` — null означает «всё ОК» |
| `TemplateService.cs` | Фабрика: `Create(name, choice, fromId, toId, catId, amount, desc)` → `TransactionTemplate` |

### 7.3. Синхронизация с сервером

| Файл | Назначение |
|------|------------|
| `SyncOrchestrator.cs` | Принимает решения о синхронизации. Фаза 1: `AnalyzeAsync()` → `SyncAnalysis` (LocalCount, ServerCount, NeedsConflictDialog при разнице > 10). Фаза 2: `ExecuteAsync(SyncAction)` → `SyncOutcome`. Не знает о UI |
| `SyncService.cs` | Низкоуровневые HTTP-операции: `SmartSyncAsync`, `SyncAsync` (pull), `PushAllDataToServerAsync`, `GetServerTransactionCountAsync` |
| `DtoMapper.cs` | Статический маппинг DTO↔Model. `FromDto(...)` — для Pull (сервер → клиент), `ToDto(...)` — для Push (клиент → сервер). Покрывает Account, Category, Obligation, Transaction, Entry |

### 7.4. Агрегация данных

#### `TransactionAggregator.cs`
Статический класс чистых функций без зависимостей от UI. Используется `ExpenseReport`, `IncomeReport`, `MonthlyReport` и `DashboardViewModel`.

| Метод | Что возвращает |
|-------|---------------|
| `FilterByDateRange(txs, from, to)` | Транзакции в диапазоне дат |
| `GetExpenseEntries(txs, accountById)` | «Расходная» проводка каждой транзакции |
| `GetIncomeEntries(txs, accountById)` | «Доходная» проводка каждой транзакции |
| `AggregateByCategoryRows(entries, ...)` | `List<CategoryShareRow>` — суммы по категориям |
| `AggregateByCategoryGroups(entries, ...)` | `List<CategoryDetailGroup>` — с детализацией по дням |
| `BuildMonthlyTotals(txs, ...)` | `List<MonthlyTotalRow>` — доходы/расходы по месяцам |
| `BuildRecentItems(txs, ...)` | `List<RecentTransactionItem>` — последние N операций |
| `BuildCurrencyBalances(accounts)` | `List<CurrencyBalance>` — балансы сгруппированные по валютам |

### 7.5. Инфраструктура

| Файл | Назначение |
|------|------------|
| `ApiService.cs` | HTTP-клиент к REST API сервера. Все запросы с Bearer-токеном |
| `AuthService.cs` | Регистрация, логин, хранение JWT в `SettingsService` |
| `SettingsService.cs` | Читает/пишет `%AppData%/Diplom/user_settings.json`. Хранит `ServerUrl`, `JwtToken`, `BaseCurrency`, `LastSyncedAt`. Событие `SettingsChanged` |
| `CurrencyRateService.cs` | Загружает курсы с сервера, сохраняет через `IDataService.SetCurrencyRate` |
| `NotificationService.cs` | Показывает диалоги: ошибки, предупреждения, подтверждение |
| `InputDialogService.cs` | Диалог ввода произвольного текста |
| `CategoryDialogService.cs` | Диалог выбора или создания категории |

## 9. Запуск приложения (пошагово)

### Шаг 1: `Program.cs`
Создаёт движок Avalonia (тема, шрифты, платформа) и запускает десктопное окно.

### Шаг 2: `App.axaml.cs`
```csharp
var win = new MainWindow { DataContext = new MainWindowViewModel() };
desktop.MainWindow = win;
```
Также регистрирует глобальные обработчики исключений (`UnhandledException`, `UnobservedTaskException`), которые показывают ошибку через `MessageDialog`.

### Шаг 3: `MainWindowViewModel` — создание сервисов
```csharp
// Слой данных
var data        = new LocalDbService();          // SQLite + репозитории
var settings    = new SettingsService();         // JSON-настройки
var notify      = new NotificationService();     // диалоги

// Сетевой слой
var api         = new ApiService(settings);
var auth        = new AuthService(settings);
var rateService = new CurrencyRateService(settings);

// Синхронизация
var syncService     = new SyncService(api, data, settings);
var syncOrchestrator = new SyncOrchestrator(syncService, data);
```

### Шаг 4: `OnWindowLoaded()`
При загрузке окна:
1. Проверка авторизации → если нет токена — диалог логина
2. Первый запуск → диалог выбора базовой валюты
3. Обновление курсов валют из интернета
4. Проверка просроченных обязательств → уведомление

## 11. Синхронизация с сервером

Вся логика решений вынесена в `SyncOrchestrator`. `AccountsViewModel.SyncAsync()` работает в два шага:

```
1. AnalyzeAsync()  →  SyncAnalysis { LocalCount, ServerCount, NeedsConflictDialog }
                              │
                    NeedsConflictDialog?
                   ┌──────── YES ─────────────────────────┐
                   │                                       │
           Показать SyncConflictDialog              SyncAction.SmartSync
           "server" → PullOnly
           "push"   → PushOnly
           "client" → Cancel
           null     → Dismiss
                   └──────────────────────────────────────┘
                              │
2. ExecuteAsync(action) → SyncOutcome { Success, DataReplaced, ErrorMessage }
                              │
                    DataReplaced? → LoadDataAsync()
```

`SmartSyncAsync` (внутри оркестратора):
- `LastSyncedAt < LocalLastChangeDate` → Push, затем Pull
- иначе → только Pull
