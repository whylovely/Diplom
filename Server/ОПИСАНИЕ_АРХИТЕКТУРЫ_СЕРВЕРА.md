# Описание архитектуры серверной части (подробно)

Серверная часть — это **веб-API**, которое работает «в фоне» и обслуживает запросы от клиентского приложения (и от веб-браузера администратора). Представьте сервер как **кухню ресторана**: клиент (посетитель) делает заказ через официанта (HTTP-запрос), кухня готовит блюдо (обрабатывает данные) и возвращает результат.

---

## Часть 1. Что за технологии используются?

### 1.1. Язык программирования — C#

Как и клиент, сервер написан на **C# (.NET 8)**. Но если клиент строит графический интерфейс (окна, кнопки), то сервер — это **консольное приложение**, которое слушает входящие HTTP-запросы и возвращает данные в формате JSON.

### 1.2. Веб-фреймворк — ASP.NET Core 8

**ASP.NET Core** — это фреймворк от Microsoft для создания веб-приложений и REST API. Он:
- Принимает HTTP-запросы (GET, POST, PUT, DELETE) от клиентов.
- Передаёт запрос в нужный **контроллер** (класс, который его обработает).
- Возвращает ответ (JSON с данными или код ошибки).

**Аналогия:** ASP.NET Core — это как **телефонная станция**. Звонок (HTTP-запрос) поступает на станцию, она смотрит номер (URL) и соединяет с нужным абонентом (контроллером).

### 1.3. База данных — PostgreSQL

Все данные (пользователи, счета, транзакции, категории, долги) хранятся в реляционной базе данных **PostgreSQL**. Это одна из самых популярных и надёжных баз данных в мире (используется в Instagram, Spotify, Netflix).

**Что значит «реляционная»?** Данные хранятся в **таблицах** (как в Excel), а между таблицами есть **связи** (relations). Например, таблица «Счета» связана с таблицей «Пользователи» через поле `UserId` — каждый счёт принадлежит конкретному пользователю.

### 1.4. ORM — Entity Framework Core 8

**ORM** (Object-Relational Mapping) — это библиотека, которая позволяет работать с базой данных **через обычные C#-классы**, без написания SQL-запросов вручную.

**Без ORM** (чистый SQL):
```sql
SELECT * FROM "Accounts" WHERE "UserId" = '...' AND "IsDeleted" = false ORDER BY "Name";
```

**С Entity Framework Core:**
```csharp
var accounts = await _db.Accounts
    .Where(x => x.UserId == userId && !x.IsDeleted)
    .OrderBy(x => x.Name)
    .ToListAsync();
```

Entity Framework **сам** превратит этот C#-код в SQL-запрос, отправит его в PostgreSQL, получит результат и превратит строки таблицы обратно в C#-объекты. Программисту не нужно знать SQL.

### 1.5. Аутентификация — JWT (JSON Web Token)

**JWT** — это способ проверки «кто вы такой?» в веб-приложениях. 

**Как это работает (пошагово):**
1. Пользователь отправляет на сервер свой email и пароль.
2. Сервер проверяет пароль. Если правильный — создаёт **токен** (длинную зашифрованную строку), в которой записано: «Это пользователь #123, с ролью User, токен действителен 7 дней».
3. Сервер возвращает токен клиенту.
4. Клиент сохраняет токен и при каждом запросе прикладывает его в заголовке: `Authorization: Bearer eyJhbGci...`
5. Сервер читает токен, расшифровывает его и понимает: «Ага, это пользователь #123, всё ок, показываю его данные».

**Зачем это нужно?** Чтобы один пользователь не мог увидеть/удалить данные другого. Каждый запрос привязан к конкретному пользователю через токен.

### 1.6. Swagger — Автоматическая документация API

**Swagger** — это веб-страница, которая автоматически генерируется из кода контроллеров и позволяет тестировать API прямо в браузере без написания клиентского кода.

Вы открываете `https://localhost:5001/swagger`, видите список всех методов, нажимаете «Try it out», вводите параметры и получаете ответ. Очень удобно для отладки и демонстрации.

### 1.7. Все библиотеки в одной таблице

| Библиотека (NuGet-пакет)                          | Для чего используется                                       |
|---------------------------------------------------|-------------------------------------------------------------|
| `Microsoft.AspNetCore` (встроено в .NET 8)        | Веб-сервер, обработка HTTP-запросов                         |
| `Npgsql.EntityFrameworkCore.PostgreSQL`            | Подключение Entity Framework к базе PostgreSQL              |
| `Microsoft.EntityFrameworkCore.Design`             | Инструменты миграции (создание таблиц в БД из кода)         |

| `System.Text.Encoding.CodePages`                   | Поддержка кодировки windows-1251 (нужно для парсинга ЦБ РФ) |

## Часть 3. Как сервер запускается (пошагово)

Весь запуск описан в одном файле — `Program.cs`. В ASP.NET Core 8 используется **Minimal API** стиль (без отдельного класса `Startup`).

### Шаг 1: Создание билдера
```csharp
var builder = WebApplication.CreateBuilder(args);
```
Создаётся «строитель» приложения, который позволяет настроить все сервисы до запуска.

### Шаг 2: Регистрация сервисов
```csharp
builder.Services.AddControllers();           // Подключаем контроллеры
builder.Services.AddHttpClient();            // HTTP-клиент для внешних запросов
builder.Services.AddMemoryCache();           // Кэш в оперативной памяти
builder.Services.AddScoped<IExchangeRateService, CbrExchangeRateService>(); // Сервис курсов
```

**Что такое `AddScoped`?** Это говорит: «когда кому-то понадобится `IExchangeRateService`, создай экземпляр `CbrExchangeRateService`». Создаётся новый экземпляр **для каждого HTTP-запроса**.

### Шаг 3: Подключение базы данных
```csharp
builder.Services.AddDbContext<AppDbContext>(opt =>
{
    opt.UseNpgsql(builder.Configuration.GetConnectionString("db"));
});
```
Строка подключения берётся из `appsettings.json`:
```json
"ConnectionStrings": {
    "db": "Host=localhost;Port=5432;Database=finance;Username=finance_user;Password=finance_pass"
}
```

### Шаг 4: Настройка JWT-аутентификации
```csharp
builder.Services
    .AddAuthentication(JwtBearerDefaults.AuthenticationScheme)
    .AddJwtBearer(opt => { ... });
```
Здесь настраивается: какой ключ используется для подписи токенов, кто их выпускает (`Issuer`), для кого они (`Audience`), и сколько допускается расхождение по времени.

### Шаг 5: Создание начальных данных (Seed)
```csharp
using (var scope = app.Services.CreateScope())
{
    // 1. Создание администратора (из appsettings.json)
    var admin = new UserEntity { Email = adminEmail, Role = "Admin" };
    // ...сохранение в БД
    
    // 2. Создание демо-пользователя с тестовыми данными
    var demoUser = new UserEntity { Email = "demo@finance.local", Role = "User" };
    // ...создаются счета, категории, операции и обязательства
    // ...сохранение в БД
}
```
При **запуске** сервер автоматически проводит миграции базы данных и создаёт:
1. Аккаунт **администратора** на основе данных из `appsettings.json`.
2. **Демо-пользователя** (`demo@finance.local` / `Demo123`) с готовым набором данных (счета, категории, операции за последний месяц, долги). Это позволяет сразу зайти в приложение и проверить работу графиков, отчётов и синхронизации без долгого ввода данных.

Этот процесс называется **seed** (засеивание начальных данных).

### Шаг 6: Запуск конвейера обработки запросов
```csharp
app.UseSwagger();           // Включаем Swagger
app.UseSwaggerUI();         // Включаем веб-интерфейс Swagger
app.UseStaticFiles();       // Раздаём файлы из wwwroot/ (админ-панель)
app.UseAuthentication();    // Проверяем JWT-токены
app.UseAuthorization();     // Проверяем права доступа
app.MapControllers();       // Привязываем маршруты к контроллерам
app.Run();                  // Запускаем сервер
```

**Порядок важен!** Сначала аутентификация (кто вы?), потом авторизация (имеете ли право?), потом контроллеры (обработка запроса).

## Часть 4. Сущности (Entities) — «Таблицы в базе данных»

Каждый файл в папке `Entities/` — это C#-класс, который соответствует **таблице** в базе данных PostgreSQL. Entity Framework автоматически создаёт таблицы на основе этих классов.

### `UserEntity` — Таблица пользователей
```csharp
public sealed class UserEntity
{
    public Guid Id { get; set; }            // Уникальный ID
    public string Email { get; set; }       // Email (логин)
    public string PasswordHash { get; set; } // Хэш пароля (НЕ сам пароль!)
    public string Role { get; set; } = "User"; // Роль: "User" или "Admin"
    public bool IsBlocked { get; set; }     // Заблокирован ли?
    public DateTimeOffset CreatedAt { get; set; } // Дата регистрации
}
```

**Почему хранится хэш, а не пароль?** Если злоумышленник украдёт базу данных, он увидит только хэши (бессмысленные строки), а не настоящие пароли. Хэширование — это **односторонний** процесс: из пароля можно получить хэш, но из хэша нельзя восстановить пароль.

### `AccountEntity` — Таблица счетов
```csharp
public sealed class AccountEntity
{
    public Guid Id { get; set; }
    public Guid UserId { get; set; }           // Чей это счёт (FK → Users)
    public UserEntity User { get; set; }       // Навигационное свойство
    public string Name { get; set; }           // Название ("Наличные", "Карта Сбербанк")
    public int Kind { get; set; }              // Тип: Актив/Доход/Расход/Долг
    public string Currency { get; set; }       // Основная валюта (RUB, USD...)
    public int AccountType { get; set; }       // 0=Обычный, 1=Многовалютный
    public string? SecondaryCurrency { get; set; } // Вторая валюта (если многовалютный)
    public decimal? ExchangeRate { get; set; }     // Курс конвертации
    public bool IsDeleted { get; set; }        // Мягкое удаление (запись НЕ удаляется из БД)
    public DateTimeOffset? DeletedAt { get; set; } // Когда удалён
}
```

**Что такое «мягкое удаление» (soft delete)?** Вместо физического удаления строки из таблицы, мы просто ставим флаг `IsDeleted = true`. Строка остаётся в БД, но фильтруется при запросах. Это позволяет:
- Восстановить данные, если пользователь передумал.
- Хранить историю для отчётов.

### `TransactionEntity` — Таблица транзакций
```csharp
public sealed class TransactionEntity
{
    public Guid Id { get; set; }
    public Guid UserId { get; set; }
    public DateTimeOffset Date { get; set; }     // Дата операции
    public string? Description { get; set; }     // Описание
    public List<EntryEntity> Entries { get; set; } // Список проводок
}
```

### `EntryEntity` — Таблица проводок
```csharp
public sealed class EntryEntity
{
    public Guid Id { get; set; }
    public Guid UserId { get; set; }
    public Guid TransactionId { get; set; }  // К какой транзакции относится
    public Guid AccountId { get; set; }      // На какой счёт
    public Guid? CategoryId { get; set; }    // В какой категории (может быть null)
    public int Direction { get; set; }       // 1=Дебет, 2=Кредит
    public decimal Amount { get; set; }      // Сумма
    public string Currency { get; set; }     // Валюта
}
```

**Почему транзакция разбита на проводки?** Это реализация принципа **двойной записи**: каждая финансовая операция затрагивает минимум два счёта. Например, «Расход 1000₽ на продукты»:
- Проводка 1: Списание 1000₽ со счёта «Наличные» (Кредит)
- Проводка 2: Зачисление 1000₽ на счёт расходов «Продукты» (Дебет)

### `ObligationEntity` — Таблица долгов
```csharp
public class ObligationEntity
{
    public Guid Id { get; set; }
    public Guid UserId { get; set; }
    public string Counterparty { get; set; }  // Кто должен / кому должен
    public decimal Amount { get; set; }       // Сумма
    public string Currency { get; set; }      // Валюта
    public int Type { get; set; }             // 0=Я должен, 1=Мне должны
    public DateTimeOffset CreatedAt { get; set; } // Дата записи
    public DateTimeOffset? DueDate { get; set; } // Дедлайн оплаты
    public bool IsPaid { get; set; }          // Оплачено ли
    public DateTimeOffset? PaidAt { get; set; } // Дата оплаты
    public string? Note { get; set; }         // Примечание
    public bool IsDeleted { get; set; }       // Soft delete
}
```

| Сущность (C#-класс) | Таблица в БД   | Что хранит                                  |
|----------------------|----------------|---------------------------------------------|
| `UserEntity`         | Users          | Пользователи (email, хэш пароля, роль)     |
| `AccountEntity`      | Accounts       | Банковские счета пользователя               |

| `ObligationEntity`   | Obligations    | Долги и обязательства                       |

## Часть 5. AppDbContext — «Карта базы данных»

Файл `AppDbContext.cs` — это центральный класс, через который Entity Framework взаимодействует с PostgreSQL. Он:

1. **Объявляет таблицы** через свойства `DbSet<>`:
```csharp
public DbSet<UserEntity> Users => Set<UserEntity>();
public DbSet<AccountEntity> Accounts => Set<AccountEntity>();
// ...
```

2. **Настраивает правила** в методе `OnModelCreating`:
   - Уникальные индексы (нельзя создать два счёта с одинаковым именем у одного пользователя).
   - Связи между таблицами (каскадное удаление, ограничения).
   - Типы данных (`decimal(18,6)` для курсов валют).
   - Глобальные фильтры (`HasQueryFilter(x => !x.IsDeleted)` — автоматически скрывает удалённые записи).

## Часть 6. Контроллеры — «Кто обрабатывает запросы»

**Контроллер** — это класс, который принимает HTTP-запрос и возвращает ответ. Каждый контроллер отвечает за свою «тему» (счета, категории, транзакции и т.д.).

### Как работает контроллер (на примере AccountsController)

```csharp
[ApiController]                        // Говорит ASP.NET: это API-контроллер

[Route("api/accounts")]                // Базовый URL: /api/accounts
public sealed class AccountsController : ControllerBase
{
    private readonly AppDbContext _db;  // Доступ к базе данных

    // GET /api/accounts — Получить все счета текущего пользователя
    [HttpGet]
    public async Task<ActionResult<List<AccountDto>>> GetAll(CancellationToken ct)
    {
        var userId = UserContext.GetUserId(User);  // Извлекаем ID из JWT
        var items = await _db.Accounts
            .Where(x => x.UserId == userId && !x.IsDeleted)
            .OrderBy(x => x.Name)
            .ToListAsync(ct);
        return Ok(items);  // Возвращаем 200 OK + JSON
    }

    // POST /api/accounts — Создать новый счёт
    [HttpPost]
    public async Task<ActionResult<AccountDto>> Create(CreateAccountRequest req, CancellationToken ct)
    {
        // Валидация → Создание Entity → Сохранение → Возврат 201 Created
    }

    // DELETE /api/accounts/{id} — Мягкое удаление (с проверкой зависимостей)
    [HttpDelete("{id:guid}")]
    public async Task<IActionResult> Delete(Guid id, [FromQuery] bool force = false, ...)
    {
        // Если есть связанные записи и force!=true → 400 BadRequest
        // Иначе → IsDeleted=true → 204 NoContent
    }
}
```

| Контроллер              | URL-префикс          | Методы                                | Описание                         |
|-------------------------|----------------------|---------------------------------------|----------------------------------|

| `ExchangeController`    | `/api/exchange`      | GET rates                             | Актуальные курсы валют           |

| `AdminController`       | `/api/admin`         | GET users, POST block, DELETE user    | Админ-панель (управление пользователями) |

### Логика авторизации

- **`[Authorize]`** на контроллере — все методы требуют JWT-токен. Без токена сервер вернёт **401 Unauthorized**.
- **`[Authorize(Policy = "Admin")]`** — метод доступен только пользователям с ролью `Admin`. Если обычный пользователь попробует — получит **403 Forbidden**.
- `UserContext.GetUserId(User)` — извлекает ID текущего пользователя из JWT-токена, чтобы показывать только **его** данные.

## Часть 7. Сервис курсов валют — «Откуда берутся курсы»

### Источники данных

Сервер получает актуальные курсы из **двух внешних API**:

1. **ЦБ РФ** (Центральный Банк России) — для фиатных валют (USD, EUR, GBP и т.д.)
   - URL: `http://www.cbr.ru/scripts/XML_daily.asp`
   - Ответ приходит в формате **XML** в кодировке **windows-1251**.
   - Сервер парсит XML, извлекает код валюты, номинал и курс.

2. **CoinGecko** — для криптовалют (BTC, ETH, USDT)
   - URL: `https://api.coingecko.com/api/v3/simple/price?ids=tether,bitcoin,ethereum&vs_currencies=usd`
   - Ответ приходит в формате **JSON**.
   - Цены в USD умножаются на текущий курс доллара ЦБ РФ.

### Кэширование (Rate-Limiting)

Чтобы не перегружать внешние API частыми запросами, сервер использует **двухуровневый кэш**:

```
┌──────────────────────────────────────────────────────────────┐
│                    Запрос: GET /api/exchange/rates            │
│                                                              │
│  1. Есть в кэше (< 1 часа)?  ──── ДА ───► Отдаём из кэша   │
│         │                                                    │
│        НЕТ                                                   │
│         │                                                    │
│  2. Запрашиваем ЦБ РФ + CoinGecko                           │
│         │                                                    │
│     Успех? ───── ДА ───► Сохраняем в кэш (1 ч + 7 дней)     │
│         │                 Отдаём свежие данные                │
│        НЕТ                                                   │
│         │                                                    │
│  3. Есть «последние известные» (< 7 дней)?                   │
│         │                                                    │
│     ДА ──► Отдаём устаревшие (лучше, чем ничего)             │
│     НЕТ ─► Возвращаем пустой список                          │
└──────────────────────────────────────────────────────────────┘
```

## Часть 9. Конфигурация (`appsettings.json`)

Все настройки сервера хранятся в файле `appsettings.json`:

```json
{
  "Jwt": {
    "Issuer": "FinanceServer",        // Кто выпускает токены
    "Audience": "FinanceClient",      // Для кого токены предназначены
    "Key": "CHANGE_THIS_TO_A_LONG..." // Секретный ключ шифрования (≥32 символа)
  },
  "Admin": {
    "Email": "admin@finance.local",   // Email администратора (seed)
    "Password": "Admin123"            // Пароль администратора (seed)
  },
  "ConnectionStrings": {
    "db": "Host=localhost;Port=5432;Database=finance;..."  // Подключение к PostgreSQL
  }
}
```

**Важно:** В продакшене (на реальном сервере) секретный ключ JWT и пароль админа должны быть заменены и храниться в переменных окружения, а не в файле.

## Часть 10. Миграции — «Как создаются таблицы в БД»

Когда вы меняете Entity (например, добавляете новое поле), нужно **обновить базу данных**. Для этого используются **миграции** Entity Framework:

```bash
# 1. Создать миграцию (описание изменений)
dotnet ef migrations add AddNewField

# 2. Применить миграцию (выполнить SQL в БД)
dotnet ef database update
```

Миграции хранятся в папке `Migrations/`. Каждая миграция — это файл с C#-кодом, который описывает, какие таблицы/колонки нужно добавить или изменить. Это как **журнал изменений** для базы данных.
